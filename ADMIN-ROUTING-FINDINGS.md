# Admin routing findings — comparison vs. a clean Payload install

Reference: `/Users/eamon/Projects/moxy-gallery` — a freshly scaffolded Payload 3 +
Next 16 app on Vercel where `/admin` works with **no** middleware/proxy and no
custom patches. This document records why `/admin` misbehaves in this project.

## Summary

The admin area in this project was **hand-scaffolded incorrectly from the first
commit** (`f92d879`), not generated by Payload's `create-payload-app` template.
Every subsequent "fix" (commits `46981c1` → `3243132`, plus `src/proxy.ts`) is a
workaround for symptoms of that incorrect scaffold, not a fix for the cause. The
clean install never needs any of them.

## The divergences (this project vs. clean install)

| Concern | Clean install (moxy-gallery) — works | This project — broken |
| --- | --- | --- |
| Route group | One `(payload)` group holds **both** admin and api | Split: hand-rolled `(admin)` group for admin, `(payload)` for api only |
| RootLayout location | `src/app/(payload)/layout.tsx` — at the **group root**, one level above `admin/` | `src/app/(admin)/admin/[[...segments]]/layout.tsx` — co-located in the deepest catch-all folder |
| `not-found.tsx` | `(payload)/admin/[[...segments]]/not-found.tsx` renders Payload's `NotFoundPage` | **Missing entirely** |
| `custom.scss` | Present, imported by layout | Missing |
| Middleware/proxy | None | `src/proxy.ts` strips `Sec-Fetch-*`, re-adds trailing-slash normalization |
| page.tsx | Stock generated `RootPage`/`generatePageMetadata` | Custom `normalizeParams` / `normalizeSearchParams` empty-segment hacks |
| Versions | next 16.2.2 / payload 3.82.0 | next 16.2.7 / payload 3.85.0 (not the cause) |

## Why this causes the symptoms

1. **Missing `not-found.tsx` is the smoking gun for the post-login 404.**
   Payload's catch-all renders its own `NotFoundPage` for paths it classifies as
   unmatched during certain RSC/document navigation cases — including the
   same-origin redirect right after login. With no `not-found.tsx`, Next falls
   back to its default 404 instead of Payload's view, which is exactly the
   "authenticated same-origin navigation 404s" behavior `src/proxy.ts` was
   written to mask.

2. **Layout co-located in the catch-all instead of the group root** changes the
   segment/layout nesting Payload's `RootLayout` expects. This is what surfaces
   the empty-segment (`segments: [""]`) case the `normalizeParams` hack and the
   proxy's trailing-slash code are fighting. In the clean install the group-root
   layout wraps the whole `/admin` subtree and the empty-segment case never
   arises.

3. **`src/proxy.ts` is self-acknowledged whack-a-mole.** Its own comments
   describe how stripping `Sec-Fetch-*` fixed the 404 but caused a redirect
   loop, so RSC requests had to be exempted (commit `3243132`). A correct
   scaffold needs none of this — the clean install ships no proxy/middleware.

## Recommended fix (restore the standard structure)

Rather than add a 4th patch, regenerate the Payload app folder to match the
clean install:

1. Recreate `src/app/(payload)/layout.tsx` (group-root RootLayout), `custom.scss`,
   `(payload)/admin/[[...segments]]/page.tsx`, and the **`not-found.tsx`** —
   copy the generated versions from moxy-gallery, fixing relative `importMap`
   paths (`./admin/importMap.js` in layout, `../importMap` in page/not-found).
2. Move the admin under `(payload)` and delete the `(admin)` group.
3. Delete `src/proxy.ts`.
4. Remove the `normalizeParams` / `normalizeSearchParams` hacks from page.tsx.
5. Re-run Payload so `importMap` regenerates in the new location, then verify
   login → `/admin` with no proxy.

Note: this is separate from the guest login flow in `src/app/api/login/route.ts`,
which queries the `participants` collection and is unrelated to the Payload admin
auth issue above.
